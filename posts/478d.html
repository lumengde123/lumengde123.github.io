一文理清Nginx-端口-反向代理等概念

卢孟德2024-02-292024-03-04

前言

本文涉及的名词：计算机网络、端口、OSI模型、HTTP、TCP、Nginx、反向代理、负载均衡

今天在学习Nginx的时候，理不清端口是什么，有什么用。所以就和ChatGPT交流，并浏览了一些文章，整理出了一个较为完整的知识块，本片文章就来记录一下学习成果。

云服务器&VPS区别

首先一台云服务器一般为一个独立的虚拟计算机实例，它还可以创建出多个虚拟机实例，即VPS（虚拟专用服务器）。每个VPS都有自己独立的操作系统、计算资源和网络配置，它们之间是相互隔离的。因此，每个VPS都可以在云服务器上独立地使用端口来提供服务，这些端口是相互隔离的，不会相互冲突。而Nginx只是个运行在VPS上的web服务器软件

端口的概念

参考文章：深入理解什么是端口(port) - 知乎 (zhihu.com)

当我们在浏览器输入域名时，其实是要通过查找DNS获取目标服务器的IP，然后根据IP:端口来访问服务器。

(IP：一个IP地址在全球范围内唯一标识一个设备，从而使设备能够相互识别和连接。)

一般来讲，浏览器会使用HTTP或HTTPS协议，至于是哪个就看域名前是用http:// 还是 https://。而这两个协议分别指定的是80和443端口。

为什么会有端口这个概念呢？首先需要了解的概念就是进程间通讯 IPC(Inter-Process Communication)。

浏览器发出请求，服务器响应请求，这其实是两个进程之间的通讯。而一台服务器上运行着非常多的进程和服务，例如web服务、ftp服务、mysql服务、Nginx进程等等。

所以浏览器的进程想要与服务器进行通讯，仅仅知道服务器IP是不够的，还需要给出进程ID，这样服务器才知道由哪个进程去响应请求。

但是为什么引入端口的概念，而不是使用进程ID呢？

简单来看有这几个原因：客户端无法知道服务端对应进程的ID、服务端对应进程重启后ID会变、每个网站的web进程ID可能不同。这就导致客户端强依赖于服务端——要与服务端通讯, 就要知道服务端对应进程的 ID。

为了解决这些问题，就引入了端口。客户端总是把请求发送到 80(或443) 端口，并反过来要求服务端去监听端口的通讯并处理请求。由此形成了反向依赖。

如果一个进程想要提供 web 服务, 它启动之后就要去绑定(binding) web 相关的端口。如果该端口被其他进程占用了，就会绑定失败。如果一个web进程没有关闭又试图启动另一个，则这两个进程都用了相同的端口，就会产生冲突。

绑定端口后就会持续监听，当有请求到来时就去响应。

端口的存在解耦了客户端与服务端之间的强依赖，整个体系变得很灵活。

以现实为例的话，可以把服务端当作市政大厅，那么端口就是一个个的服务窗口。

从客户端的角度：

  • 80 443窗口就是专门处理HTTP/HTTPS请求相关的业务。类似的，常见情况下22窗口用于处理SSH传输业务，21窗口处理FTP文件传输业务，3306用来处理MySQL的连接和数据交互业务。

从服务端角度：

  • 窗口默认是空的，如果想要提供哪些服务，那就需要运行相应的进程来占用窗口，处理业务。例如，Nginx软件监听服务器的80 443端口以提供web服务，MySQL软件监听3306端口来提供跟MySQL数据库有关的服务。在这里，Nginx和MySQL可以理解为窗口的工作人员。

Nginx介绍

Nginx是一款流行的web服务器软件。Nginx 既是软件，也是进程，同时也可以被称为服务器。

  • 软件：作为软件，Nginx 提供了一系列的功能和特性，包括但不限于 HTTP 服务器、反向代理、负载均衡、缓存加速、SSL/TLS 加密、虚拟主机、HTTP/2 支持等。Nginx 的软件版本通常包含了可执行文件和配置文件等组成部分。

  • 进程：当Nginx被启动时，操作系统会创建一个 Nginx 进程（通常是 master 进程）），该进程会加载 Nginx 的配置文件，并启动多个工作进程（通常是 worker 进程）来处理客户端的请求。每个 Nginx 进程都是操作系统中正在运行的程序的实例，拥有自己的代码、数据、堆栈等信息。

  • 服务器：Nginx 也可以被称为服务器，这是因为它提供了 HTTP 服务器的功能，能够接收客户端的 HTTP 请求，并返回相应的 HTTP 响应。即可以处理客户端和服务器之间的通信

如果把Linux当作市政大厅的话，端口可以理解为每个服务窗口。窗口里面的工作人员就是提供相应服务的应用程序。

由于Linux本身不提供web服务，所以需要安装web服务器软件如Nginx、Apache等，才可以作为服务器提供web服务。

当安装好Nginx并启动后，Nginx会监听80 443端口，可以理解为Nginx成为了80 443窗口的工作人员。一个端口只有一个服务能监听，否则会冲突。Nginx可以创建多个worker进程，所以Nginx在80/443端口可以监听很多个网站的请求。

当有客户端向本机发送HTTP/HTTPS请求时，会通过80 443端口，就会被Nginx监听到，并根据其配置文件执行相应操作。

浏览器访问域名过程

当你在浏览器输入网址时，就能得到网页文件并加载出页面，那在这过程中发生了什么呢？

域名解析（DNS解析）：

  • 点击域名后，浏览器首先会检查自己的DNS缓存，看是否已经解析过这个域名对应的IP地址。如果有，就直接使用缓存的IP地址。
  • 如果没有缓存或缓存过期，浏览器会向本地系统的DNS解析器发送一个DNS查询请求，以获取该域名对应的IP地址。
  • 本地系统的DNS解析器会查询本地缓存、hosts文件，如果没有找到，就会向互联网上的DNS服务器发送DNS查询请求。

DNS解析过程：

  • DNS服务器会按照递归查询或迭代查询的方式，依次向上级DNS服务器查询，从根域名服务器->顶级域名服务器->权威DNS服务器。
  • 权威DNS服务器会返回该域名对应的IP地址给本地DNS解析器，然后本地DNS解析器将IP地址返回给浏览器

建立TCP连接：

  • 一旦浏览器获得了目标服务器的 IP 地址，就会使用解析得到的IP地址和HTTP默认端口（80）或者HTTPS默认端口（443），向服务器发起一个TCP连接请求（通过三次握手），请求建立一个与目标服务器的连接。【如果是HTTPS请求，浏览器还需要进行SSL/TLS握手过程来建立安全连接。】

发送 HTTP 请求：

  • 一旦 TCP 连接建立成功，浏览器就会构造HTTP请求报文，其中包含请求的方法（GET、POST等）、目标URL、请求头部等信息。
  • 通过该TCP连接，浏览器将构造好的一个HTTP请求报文发送给目标服务器。这个 HTTP 请求包含了你想要获取的资源的信息，比如网页文件、图片、视频等。

服务器响应：

  • 目标服务器收到了浏览器发送的 HTTP 请求后，会根据请求的内容进行处理，并生成一个 HTTP 响应。这个响应包含了请求的资源以及一些其他的信息，比如状态码、头部信息等。

接收响应：

  • 一旦服务器生成了 HTTP 响应，它会通过 TCP 连接发送给浏览器。浏览器接收到了 HTTP 响应后，会解析响应内容，并根据内容显示网页或执行其他操作。
  • 例如浏览器获取了 index.html 页面，就会解析该页面的 HTML 内容，如果发现页面中引用了其他资源（比如 JavaScript 文件、CSS 文件、图片等），则会根据这些链接向服务器发起额外的 HTTP 请求来获取这些资源。
  • 这些额外的 HTTP 请求是在同一个 TCP 连接上进行的，因为 TCP 连接是持久的，在该连接上可以发送多个 HTTP 请求和接收多个 HTTP 响应。
  • 所以浏览器多次向服务器发送HTTP请求来获取文件。服务器则多次响应。直到最后完全加载完这个文件并进行页面渲染，最终呈现给用户。

关闭连接：

  • 当一个网页中的所有资源都加载完毕后，浏览器会保持 TCP 连接开启一段时间，以便处理可能的后续请求，这个时间被称为持久连接（Keep-Alive）。
  • 如果在这段时间内，用户点击了页面上的按钮或者进行了其他交互操作，浏览器会根据操作构造并发起相应的 HTTP 请求。如果这些请求涉及到新的资源（比如新的页面、图片、脚本等），浏览器会在同一个 TCP 连接上发送这些请求，并继续保持连接。
  • 只有当一段时间内没有新的 HTTP 请求发送到服务器，或者服务器主动关闭连接，或者达到了连接持续时间的最大限制时，浏览器才会关闭 TCP 连接。
  • 持久连接会持续一段时间，目的是为了减少后续请求的延迟，提高页面加载速度。但不会持续太久，浏览器会在一定时间后自动关闭未使用的持久连接并释放资源，目的是节约服务器资源。

VPS处理浏览器请求过程

从VPS/服务器角度来看，发生了什么呢？

这里以Django网站项目为例。先来讲讲VPS上的Django项目关联哪些软件：

首先Django项目会放在/var/www/html站点文件目录，web服务器为Nginx接受HTTP/HTTPS，然后反代到WSGI 服务器(作为后端服务器)，在WSGI 服务器( Gunicorn 或 uWSGI)中启动了Django应用程序，配置settings.py连接到了MySQL数据库。

当Nginx监听80 443端口，接收到HTTP/HTTPS请求后：

查找匹配的服务器块：

  • Nginx 根据请求的域名和请求的 URI 在配置文件中查找匹配的服务器块（server block）。

例如：请求域名为https://blog.lumengde.com/depart/list ，则在Nginx监听443端口server_name为blog.lumengde.com的server块中找到匹配/depart/list的location服务块。

处理请求_转发请求给后端应用程序：

  • 在示例配置中，包含了反向代理指令 proxy_pass，将 HTTPS 请求转发到了 http://127.0.0.1:8000，这意味着后端 Django 应用程序在本地的 8000 端口上监听请求。Nginx 根据该指令将请求转发给后端应用程序（如 Gunicorn 或其他 WSGI 服务器）处理。

后端应用程序处理请求：

  • 后端应用程序（Django 项目）接收到请求后，根据请求的 URI 和其他信息执行相应的处理逻辑，可能涉及到路由、视图函数等。
  • Django 项目通常使用 WSGI 服务器（如 Gunicorn）来监听本地的端口，并处理传入的请求。后端应用程序会根据 URL 路由和视图函数等逻辑来处理请求，执行相应的业务逻辑、查询数据库等操作。
  • 这里传入的URL请求/depart/list会在 urls.py 的path中匹配到对应的view视图函数，在view.py中视图函数会执行业务逻辑处理请求、访问数据库等等。

生成响应：

  • 后端应用程序处理完请求后，生成 HTTPS 响应，包括所需的 HTML 内容、状态码等。

返回响应给 Nginx：

  • 后端应用程序将生成的 HTTP 响应返回给 Nginx，通常是通过本地的某个端口，例如 8000 端口。

Nginx 返回响应给客户端：

  • Nginx 接收到来自后端应用程序的 HTTPS 响应后，将其返回给客户端。浏览器接收到响应后，根据响应的内容进行页面渲染或其他相应操作。至此，完成了整个请求-响应周期。

反向代理

参考文章：

  • 终于有人把正向代理和反向代理解释的明明白白了！-腾讯云开发者社区-腾讯云 (tencent.com)
  • Nginx配置反向代理，一篇搞定！ - 知乎 (zhihu.com)

先来说下什么是正向代理：

简单举个例子，你想在国内访问Youtube，但是访问被长城防御挡下了。如果你以既能连接国内网络也能连接国外网络的服务器作为”中转”，那么就能顺利访问国外的网站。所谓的搭梯子也是这个原理。这中间用来做”跳板”的服务器，就叫做正向代理。

而Nginx除了能提供HTTP/HTTPS服务之外，还能进行反向代理。

  • 反向代理是Nginx的一个重要功能，它允许Nginx接收客户端的请求，并将这些请求转发到后端的真实服务器(不一定是本机)，然后将后端服务器的响应返回给客户端。

  • 反向代理的好处是隐藏了后端服务器的真实信息，提供了负载均衡、缓存、安全过滤等功能，同时也增强了网站的可靠性和安全性。

正向代理和反向代理的区别：

  • 正向代理其实是客户端的代理，帮助客户端访问其无法访问的服务器资源。反向代理则是服务器的代理，帮助服务器做负载均衡，安全防护等。

  • 正向代理中，服务器不知道真正的客户端到底是谁，以为访问自己的就是真实的客户端。而在反向代理中，客户端不知道真正的服务器是谁，以为自己访问的就是真实的服务器。

  • 正向代理和反向代理的作用和目的不同。正向代理主要是用来解决访问限制问题。而反向代理则是提供负载均衡、安全防护等作用。二者均能提高访问速度。

  • 正向代理一般是客户端架设的，比如在自己的机器上安装一个代理软件。而反向代理一般是服务器架设的，比如在自己的机器集群中部署一个反向代理服务器。
